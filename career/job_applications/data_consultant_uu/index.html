Research Data Consultant (Data Steward) at Utrecht University

Preparing the first interview

Possible questions

Technical

  • You implemented solutions for a broad variety of data related problems, could you give us some examples?

    GUIDELINE: talk about your experience with data reduction from satellites;

    I always worked with data from satellites. These data are usually publicly available online. The data are organised according to target, the source that the satellite observes, and observations, a continous interval of time that the satellite spent observing that target.

    Therefore, these data have to be first downloaded and stored locally. Each satellite has different characteristics, for example some orbit quite far from Earth while another, in particular, is attached to the international space station. So, according to the satellite, data have to be cleaned and filtered.

    Then data need to be reduced into a higher level data, the ones that it's actually needed to perform research, and analysed. Usually I perform a preliminary analysis to check if something interesting showing up and also a deeper statistical analysis on specic cases. For both these phases visualization is fundamental as well.

    So for broad variety of problems I mean indeed downloading, storing, cleaning, filtering, reducing, analysing and creating higher level data, designing a format for this high level data, designing their corresponding meta_data, and quickly visualising these data as well in different formats (an interactive plot, a pdf etc.)

    In terms of structure also it is necessary to find the most simple storing structures for a certain data set;

  • How do you deal with large databases?

    This of course depends on a specific case, but in general I would try to access first how big or how big is going to be the dataset, the kind of data to be stored (tables, images, etc), and some other information about accessing the data, for example how many times the data need to be accessed during the research project, which kind of speed is required, and which kind of analysis will be perfomed and how.

    The next step would be trying to find optimal solutions for these database characteristics, always keeping an eye at the cost of these characteristics, and then presenting and explaining pro and cons of these solutions to the researcher.

    From a more technical point of view, I would suggest to make a copy of the raw data, so that the original raw data cannot be manipulated directly. It is also important to keep track of all the operations perfomed on raw data, this can be done via metadata or using version control software for files. The principle is the same of git but these work for files, Zenodo and Dat are possible technologies.

    Creating optimal metadata is very also important, when writing metadata I always think them as messages for my future self and my future self is a dumper and slower version of my present self and he forgot everything about the database, so it has to be information structured in a way to be easely interpreted without previous knowledge of the dataset structure.

    Every analysis also should be done in a self contained computing environment (using docker container or any kind of virtual environment for example) in a way that not only you can reproduce the same result using the same software, but also, eventually, figuring out the cause of possible errors and mistakes.

    Useful concepts:
    • Table normalization: Normalization is the process of organizing data in a database. This includes creating tables and establishing relationships between those tables according to rules designed both to protect the data and to make the database more flexible by eliminating redundancy and inconsistent dependency;
  • How did you make your data FAIR?

    I try to be sure that the raw data used for my research project are publicly available and I give detailed description on which dataset has been used. I also make detailed documentation of all the operations perfomed on data and I make available on internet the higher level data produced and the software that has been used;

  • How do you keep yourself updated?

    I am subscribed to several newsletter reguarding data science, software developement, and in general the tech world. For example TLDR and pycoder weekly are among my favourite. I like TLDR because it organises articles in terms of the time you are supposed to spend to read it. I am also memeber of facebook groups related to the data science world and I can follow up the news from there. I also follow website like kaggle, medium.com, and towardsdatascience.com.

  • Have you ever written a data management plan? Can you descrive an effective data management plan?

    I would first check if there is a data management plan tool made by the institution I am working in or I am applying fundings for.

    Said that, I would try to specify the following information:

    Project, experiment, and data description
    • Purpose of research;
    • type of data;
    • how much data will be generated in this research;
    • how long will the data be collected and how often will it change;
    • if the researcher will use data collected from someone else;
    • who is responsible for managing the data.
    Documentation, organization, and storage
    • what documentation will be created in order to make the data understandable by other researchers;
    • Use of metadata, checking if there is a standard in the field and how metadata will be stored;
    • file formats to be used;
    • naming conventions;
    • local storagen and backup procedures;
    • tools and software required to read and visualise data.
    Access, sharing, and re-use
    • who has the right to manage the data;
    • what data will be shared, when, and how?
    • if data sharing raises privacy, ethical, or confidentiality concerns?
    • who holds intellectual property rights for the data;
    • if there is any patent or technology-licensing-related restrinctions on data;
    • if the research will be published;
    • possible embargoes on data;
    • permission to re-use data and tools.
    Archiving
    • the way data will be archived and for how long;
    • preparing data for preservation and sharing;
    • if software is needed to use the data, specify hot it is going to be archived;
    • data retaining time.

  • Which kind of technical advice did you give to students and biology researchers?

    I gave both research and technical advice. On the technical advice I alway tried to understand first which kind of data they were going to handle and if they were going to collect the data theirself or if they were going to use data from other people. Then I tried to provide different solutions discussion pro and cons.

  • Can you share an effective approach for dealing with large amount of data?

    In few words:

    • create a structure for the research data. This includes being able to access the data in a readable format;
    • always make a backup of your raw data;
    • try to plan ahead figuring out future size, data type and use of your database;
    • run any operation in a virtual environment so that in ten years it will be possible to perform the same operations in the same conditions of today;
    • track any performed operations with detailed metadata and version control systems;
    • if not necessary, perform operations on cloud, do not download the data;

  • What is in your opinion the most difficoult part of dealing with big data?

    Thinking long-term, in particular estimating future size and use of a database. Database can grow very complex with time and with the rapid change of technology, designing database effective solutions that can last over time is, in my opinion, the greatest challenge;

  • What is in your opinion the worst thing that can happen in data management?

    Data corruption or damage that goes under the radar and it is not noticed by the researcher. That would mean that scientific results will be based on false evidence and that is worse than loosing completely the data;
